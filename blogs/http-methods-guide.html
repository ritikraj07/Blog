HTTP Methods Overview

Complete reference for all HTTP/1.1 request methods with deep technical explanations

What are HTTP Methods?

HTTP methods (also called HTTP verbs) indicate the desired action to be performed on the identified resource. They form the core of RESTful API design and web communication.

Key HTTP Concepts
  • Safe Methods: GET, HEAD, OPTIONS - No server state modification
  • Idempotent Methods: GET, PUT, DELETE, HEAD, OPTIONS - Same effect when called multiple times
  • Cacheable Methods: GET, HEAD - Responses can be cached
  • Request Body Allowed: POST, PUT, PATCH, CONNECT

HTTP Method Evolution

HTTP methods have evolved through different specifications:

  • HTTP/0.9 (1991): Only GET method
  • HTTP/1.0 (1996): GET, HEAD, POST
  • HTTP/1.1 (1997): GET, HEAD, POST, PUT, DELETE, TRACE, OPTIONS, CONNECT
  • RFC 5789 (2010): PATCH method added
  • HTTP/2 (2015), HTTP/3 (2022): Same methods, different transport

GET Method - Retrieve Resources

The fundamental method for safe resource retrieval

GET
Retrieve a representation of the specified resource. Requests using GET should only retrieve data.
Safe
Idempotent
Cacheable

What is GET?

GET is the primary method for retrieving resource representations. It's safe (no side effects), idempotent, and cacheable.

Why is GET Safe and Idempotent?

Safe: GET requests should not modify any server state. They are read-only operations.

Idempotent: Multiple identical GET requests return the same response (assuming resource hasn't changed).

Important Considerations

While GET should be safe, some implementations might have side effects (like logging). However, according to HTTP specification, GET should be safe.

How do I use GET?

GET requests typically use query parameters for filtering and should never have a request body.

GET with Query Parameters 
# Get users with pagination and filtering
curl -X GET "https://api.example.com/users?page=1&limit=10&active=true" \
  -H "Accept: application/json" \
  -H "Authorization: Bearer token123"

# Get specific user by ID
curl -X GET "https://api.example.com/users/123" \
  -H "Accept: application/json"

# Search with complex parameters
curl -X GET "https://api.example.com/search?q=javascript&type=article&sort=date&order=desc" \
  -H "Accept: application/json"
GET with Fetch API 
// Get users with query parameters
async function getUsers(page = 1, limit = 10) {
  const queryParams = new URLSearchParams({
    page: page.toString(),
    limit: limit.toString(),
    active: 'true'
  });
  
  try {
    const response = await fetch(`https://api.example.com/users?${queryParams}`, {
      method: 'GET',
      headers: {
        'Accept': 'application/json',
        'Authorization': 'Bearer token123'
      }
    });
    
    if (response.ok) {
      const users = await response.json();
      return users;
    } else {
      throw new Error(`HTTP ${response.status}: ${response.statusText}`);
    }
  } catch (error) {
    console.error('GET request failed:', error);
    throw error;
  }
}

// Usage
getUsers(1, 10)
  .then(users => console.log('Users:', users))
  .catch(error => console.error('Error:', error));
GET with Python Requests 
import requests

def get_users(page=1, limit=10, active=True):
    """GET request with query parameters"""
    try:
        params = {
            'page': page,
            'limit': limit,
            'active': str(active).lower()
        }
        
        response = requests.get(
            'https://api.example.com/users',
            params=params,
            headers={
                'Accept': 'application/json',
                'Authorization': 'Bearer token123'
            }
        )
        
        response.raise_for_status()  # Raises exception for 4xx/5xx status
        return response.json()
        
    except requests.exceptions.RequestException as e:
        print(f"GET request failed: {e}")
        raise

# Usage
try:
    users = get_users(page=1, limit=10)
    print(f"Retrieved {len(users)} users")
except Exception as e:
    print(f"Error: {e}")

POST Method - Create Resources

The flexible method for creating resources and performing actions

POST
Submit an entity to the specified resource, often causing a change in state or side effects
Unsafe
Not Idempotent
Not Cacheable

Why is POST Not Idempotent?

POST is not idempotent because multiple identical requests typically create multiple resources. This is by design for operations where duplication matters.

Real-world Example

If you POST a payment request twice, you might get charged twice. This is why payment processors use idempotency keys with POST requests.

How do I use POST?

POST requests include a request body and are used for creation and complex operations.

POST Request Examples

PUT Method - Replace Resources

The idempotent method for complete resource replacement

PUT
Replace all current representations of the target resource with the request payload
Unsafe
Idempotent
Not Cacheable

Why is PUT Idempotent?

PUT is idempotent because multiple identical requests have the same effect as a single request. This makes PUT ideal for:

  • Reliable updates - Can be safely retried on network failures
  • Concurrent operations - Last write wins semantics
  • Distributed systems - Consistent behavior across retries

PUT vs POST for Creation

POST for creation: Server generates resource ID
PUT for creation: Client specifies resource ID

PUT Request Examples 
# CREATE with PUT (client controls ID)
curl -X PUT "https://api.example.com/users/alice-johnson" \
  -H "Content-Type: application/json" \
  -d '{
    "name": "Alice Johnson",
    "email": "alice@example.com"
  }'
# Response: 201 Created (if new) or 200 OK (if updated)

# UPDATE with PUT (complete replacement)
curl -X PUT "https://api.example.com/users/123" \
  -H "Content-Type: application/json" \
  -d '{
    "id": 123,
    "name": "Alice Smith",
    "email": "alice.smith@example.com",
    "age": 30,
    "preferences": {"theme": "dark"}
  }'
# Note: Missing fields from previous state are removed!

PATCH Method - Partial Updates

The efficient method for updating specific resource fields

PATCH
Apply partial modifications to a resource without replacing the entire entity
Unsafe
Not Idempotent*
Not Cacheable

PATCH Formats and Idempotency

PATCH idempotency depends on the patch format used:

PATCH Request Examples 
# JSON Merge Patch (RFC 7396) - Not Idempotent
curl -X PATCH "https://api.example.com/users/123" \
  -H "Content-Type: application/merge-patch+json" \
  -d '{
    "age": 31,
    "preferences": {
      "theme": "light"
    }
  }'

# JSON Patch (RFC 6902) - Idempotent
curl -X PATCH "https://api.example.com/users/123" \
  -H "Content-Type: application/json-patch+json" \
  -d '[
    { "op": "replace", "path": "/age", "value": 31 },
    { "op": "replace", "path": "/preferences/theme", "value": "light" },
    { "op": "add", "path": "/tags", "value": ["vip"] },
    { "op": "remove", "path": "/phone_number" }
  ]'

# Custom operations format
curl -X PATCH "https://api.example.com/users/123" \
  -H "Content-Type: application/json" \
  -d '{
    "$set": {
      "age": 31,
      "preferences.theme": "light"
    },
    "$inc": {
      "login_count": 1
    }
  }'

When to Use PATCH vs PUT

PATCH vs PUT Decision Guide

Use PUT when:
You're replacing the entire resource
You have all the resource data
You need strong idempotency guarantees
Bandwidth is not a concern

Use PATCH when:
You're updating specific fields only
You want to conserve bandwidth
Multiple clients might update different fields
The resource is large with many optional fields

DELETE Method - Remove Resources

The idempotent method for resource removal

DELETE
Delete the specified resource from the server
Unsafe
Idempotent
Not Cacheable

Why is DELETE Idempotent?

DELETE is idempotent because:

  • First DELETE: Resource is removed, returns 200 OK or 204 No Content
  • Subsequent DELETE: Resource already gone, returns 404 Not Found or 410 Gone
  • Same result: Resource doesn't exist after first call
DELETE Request Examples 
# Delete a user
curl -X DELETE "https://api.example.com/users/123" \
  -H "Authorization: Bearer token123"

# Response: 204 No Content (success, no body)
# or 200 OK with deletion confirmation

# Subsequent request to same resource:
curl -X DELETE "https://api.example.com/users/123" \
  -H "Authorization: Bearer token123"

# Response: 404 Not Found or 410 Gone
# Both indicate resource doesn't exist - idempotent!
DELETE with Fetch API 
// Delete resource with DELETE
async function deleteResource(resourceUrl) {
  try {
    const response = await fetch(resourceUrl, {
      method: 'DELETE',
      headers: {
        'Authorization': 'Bearer token123'
      }
    });
    
    // DELETE is idempotent - 404 is considered success for idempotency
    if (response.ok || response.status === 404 || response.status === 410) {
      return {
        success: true,
        status: response.status,
        message: response.status === 404 ? 'Resource already deleted' : 'Resource deleted'
      };
    } else {
      throw new Error(`HTTP ${response.status}: ${response.statusText}`);
    }
  } catch (error) {
    console.error('DELETE request failed:', error);
    throw error;
  }
}

// Batch delete multiple resources
async function deleteMultiple(resourceUrls) {
  const results = [];
  
  for (const url of resourceUrls) {
    try {
      const result = await deleteResource(url);
      results.push({ url, ...result });
    } catch (error) {
      results.push({ url, success: false, error: error.message });
    }
  }
  
  return results;
}

// Usage
deleteResource('https://api.example.com/users/123')
  .then(result => {
    console.log('Delete result:', result);
  })
  .catch(error => console.error('Error:', error));
DELETE with Python Requests 
import requests

def delete_resource(resource_url):
    """DELETE request to remove a resource"""
    try:
        response = requests.delete(
            resource_url,
            headers={'Authorization': 'Bearer token123'}
        )
        
        # DELETE is idempotent - 404 is considered success for idempotency
        if response.ok or response.status_code in [404, 410]:
            return {
                'success': True,
                'status': response.status_code,
                'message': 'Resource already deleted' if response.status_code == 404 else 'Resource deleted'
            }
        else:
            response.raise_for_status()
            
    except requests.exceptions.RequestException as e:
        print(f"DELETE request failed: {e}")
        raise

def delete_multiple(resource_urls):
    """Batch delete multiple resources"""
    results = []
    
    for url in resource_urls:
        try:
            result = delete_resource(url)
            results.append({'url': url, **result})
        except Exception as e:
            results.append({'url': url, 'success': False, 'error': str(e)})
    
    return results

# Usage
try:
    result = delete_resource('https://api.example.com/users/123')
    print('Delete result:', result)
except Exception as e:
    print(f"Error: {e}")

DELETE Response Codes

Different DELETE response codes and their meanings:

  • 200 OK - Deletion successful, response body may contain status
  • 202 Accepted - Deletion queued for processing
  • 204 No Content - Deletion successful, no response body
  • 404 Not Found - Resource doesn't exist (treated as success for idempotency)
  • 410 Gone - Resource previously existed but is now gone

Soft Delete vs Hard Delete

Implementation Considerations

Soft Delete: Mark resource as deleted but keep in database
DELETE /users/123 sets deleted_at timestamp

Hard Delete: Permanently remove resource from database
DELETE /users/123 removes database record

Best Practice: Use soft delete for user data, hard delete for temporary data. Always consider data recovery requirements.

HEAD Method - Resource Metadata

Retrieve resource headers without the response body

HEAD
Identical to GET but returns only headers, no response body
Safe
Idempotent
Cacheable

Why Use HEAD Instead of GET?

HEAD is optimized for efficiency when you only need metadata:

  • Check if resource exists without downloading content
  • Get Content-Length before deciding to download large files
  • Check Last-Modified for caching decisions
  • Validate links without transferring content
  • Monitor API health with minimal bandwidth

HEAD Implementation Examples

HEAD Request Examples 
# Check if resource exists and get metadata
curl -I "https://api.example.com/users/123" \
  -H "Authorization: Bearer token123"

# Response headers only:
# HTTP/1.1 200 OK
# Content-Type: application/json
# Content-Length: 245
# Last-Modified: Wed, 21 Oct 2015 07:28:00 GMT
# ETag: "abc123"
# Cache-Control: max-age=3600

# Check file size before download
curl -I "https://api.example.com/files/document.pdf"

# You can decide to download based on size
// Check resource metadata with HEAD
async function checkResource(url) {
  try {
    const response = await fetch(url, {
      method: 'HEAD',
      headers: {
        'Authorization': 'Bearer token123'
      }
    });
    
    if (response.ok) {
      const metadata = {
        exists: true,
        contentLength: response.headers.get('Content-Length'),
        contentType: response.headers.get('Content-Type'),
        lastModified: response.headers.get('Last-Modified'),
        etag: response.headers.get('ETag')
      };
      return metadata;
    } else if (response.status === 404) {
      return { exists: false };
    } else {
      throw new Error(`HTTP ${response.status}`);
    }
  } catch (error) {
    console.error('HEAD request failed:', error);
    throw error;
  }
}

// Check multiple resources efficiently
async function checkResources(urls) {
  const results = [];

    try {
      const metadata = await checkResource(url);
      results.push({ url, ...metadata });
    } catch (error) {
      results.push({ url, error: error.message });
    }
  }
  
  return results;
}

            }
        elif response.status_code == 404:
            return {'exists': False}
        else:
            response.raise_for_status()
            
    except requests.exceptions.RequestException as e:
        print(f"HEAD request failed: {e}")
        raise

def check_resources(urls):
    """Check multiple resources efficiently"""
    results = []
    
    for url in urls:
        try:
            metadata = check_resource(url)
            results.append({'url': url, **metadata})
        except Exception as e:
            results.append({'url': url, 'error': str(e)})
    
    return results

# Usage
metadata = check_resource('https://api.example.com/users/123')
if metadata['exists']:
    print(f"Resource size: {metadata['content_length']} bytes")

// Discover API capabilities with OPTIONS
async function discoverApi(url) {
  try {
    const response = await fetch(url, {
      method: 'OPTIONS',
      headers: {
        'Accept': 'application/json'
      }
    });
    
    if (response.ok) {
      const allowHeader = response.headers.get('Allow');
      const capabilities = {
        allowedMethods: allowHeader ? allowHeader.split(', ').map(m => m.trim()) : [],
        cors: {
          allowOrigin: response.headers.get('Access-Control-Allow-Origin'),
          allowMethods: response.headers.get('Access-Control-Allow-Methods'),
          allowHeaders: response.headers.get('Access-Control-Allow-Headers')
        }
      };
      
      // If response has body, parse it for additional info
      if (response.status !== 204) {
        try {
          capabilities.details = await response.json();
        } catch (e) {
          // No JSON body or parsing failed
        }
      }
      
      return capabilities;
    } else {
      throw new Error(`HTTP ${response.status}: ${response.statusText}`);
    }
  } catch (error) {
    console.error('OPTIONS request failed:', error);
    throw error;
  }
}

async function simulateCorsPreflight(url, origin, method, headers) {

    const requestHeaders = {
      'Origin': origin,
      'Access-Control-Request-Method': method
    };
    
    if (headers && headers.length > 0) {
      requestHeaders['Access-Control-Request-Headers'] = headers.join(', ');

      headers: requestHeaders
    });
    
    return {
      allowed: response.ok,
      headers: {
        allowOrigin: response.headers.get('Access-Control-Allow-Origin'),
        allowMethods: response.headers.get('Access-Control-Allow-Methods'),
        allowHeaders: response.headers.get('Access-Control-Allow-Headers')
      }
    };
  } catch (error) {
    console.error('CORS preflight simulation failed:', error);
    throw error;
  }
}

CONNECT Method - Tunnel Establishment

Establish a network tunnel to the target resource

CONNECT
Establish a tunnel to the server identified by the target resource
Unsafe
Not Idempotent
Not Cacheable
Specialized

What is CONNECT Used For?

CONNECT is primarily used for SSL/TLS tunneling through HTTP proxies:

  • HTTPS through corporate proxies
  • SSL termination at proxy level
  • Establishing secure tunnels
  • Proxy chaining and forwarding
Security Implications

CONNECT can be used to bypass security controls. Proper proxy configuration should restrict CONNECT to specific ports (typically 443 for HTTPS).

CONNECT in Action

CONNECT Through HTTP Proxy 
# Establish SSL tunnel through proxy to api.example.com:443
curl -x proxy.example.com:8080 \
  --proxy-anyauth \
  --proxy-user username:password \
  -X CONNECT api.example.com:443

# Once tunnel established, regular HTTPS requests flow through:
curl -x proxy.example.com:8080 \
  --proxy-anyauth \
  --proxy-user username:password \
  https://api.example.com/users
CONNECT with Node.js 
// Node.js HTTP proxy server handling CONNECT requests
const http = require('http');
const net = require('net');
const { URL } = require('url');

// Create proxy server
const proxy = http.createServer();

proxy.on('connect', (req, clientSocket, head) => {
  // Extract target host and port from CONNECT request
  const { port, hostname } = new URL(`http://${req.url}`);
  
  console.log(`CONNECT request to ${hostname}:${port}`);
  
  // Create connection to target server
  const serverSocket = net.connect(port || 443, hostname, () => {
    // Send HTTP 200 to client when tunnel is established
    clientSocket.write('HTTP/1.1 200 Connection Established\r\n\r\n');
    
    // Pipe the sockets together
    serverSocket.write(head);
    serverSocket.pipe(clientSocket);
    clientSocket.pipe(serverSocket);
  });
  
  // Handle errors
  serverSocket.on('error', (err) => {
    console.error('Target server error:', err);
    clientSocket.end('HTTP/1.1 500 Connection Failed\r\n\r\n');
  });
  
  clientSocket.on('error', (err) => {
    console.error('Client socket error:', err);
    serverSocket.end();
  });
});

// Start proxy server
proxy.listen(8080, () => {
  console.log('HTTP proxy server listening on port 8080');
});

CONNECT with Python 
import socket
import ssl
import requests
from urllib.parse import urlparse

class HTTPProxyConnector:
    """HTTP proxy connector with CONNECT support"""
    
    def __init__(self, proxy_host, proxy_port):
        self.proxy_host = proxy_host
        self.proxy_port = proxy_port
    
    def connect_via_proxy(self, target_host, target_port=443):
        """Establish CONNECT tunnel through proxy"""
        try:
            # Connect to proxy
            proxy_sock = socket.socket(socket.AF_INET, socket.SOCK_STREAM)
            proxy_sock.connect((self.proxy_host, self.proxy_port))
            
            # Send CONNECT request
            connect_req = f"CONNECT {target_host}:{target_port} HTTP/1.1\r\nHost: {target_host}:{target_port}\r\n\r\n"
            proxy_sock.send(connect_req.encode())
            
            # Read proxy response
            response = proxy_sock.recv(4096).decode()
            if not response.startswith('HTTP/1.1 200'):
                raise Exception(f"Proxy connection failed: {response}")
            
            # Wrap socket with SSL for HTTPS
            context = ssl.create_default_context()
            ssl_sock = context.wrap_socket(proxy_sock, server_hostname=target_host)
            
            return ssl_sock
            
        except Exception as e:
            print(f"CONNECT tunnel failed: {e}")
            raise

def make_request_via_proxy(target_url, proxy_url):
    """Make request through HTTP proxy"""
    try:
        proxies = {
            'http': proxy_url,
            'https': proxy_url
        }
        
        response = requests.get(target_url, proxies=proxies, verify=True)
        return response
        
    except requests.exceptions.RequestException as e:
        print(f"Request via proxy failed: {e}")

CONNECT vs Other Methods

CONNECT Unique Characteristics
  • Not a REST method - Doesn't operate on resources
  • Creates bidirectional tunnel - Unlike request-response pattern
  • Used by clients, not applications - Typically browser/proxy communication
  • Requires special proxy support - Not all HTTP servers handle CONNECT

TRACE Method - Request Diagnostics

Perform a message loop-back test for debugging

TRACE
Perform a message loop-back test along the path to the target resource
Safe
Idempotent
Not Cacheable
Diagnostic

What is TRACE Used For?

TRACE is primarily a diagnostic tool used for:

  • Debugging proxy chains - See how requests are modified
  • Testing request transformations - Headers, content modifications
  • Security testing - Identifying request manipulation
  • Network troubleshooting - Path verification
Security Warning

TRACE can be used in Cross-Site Tracing (XST) attacks to steal cookies. It should be disabled on production servers.

TRACE Examples

TRACE Request Example 
# Perform TRACE request
curl -X TRACE "https://api.example.com/users" \
  -H "Custom-Header: value" \
  -H "User-Agent: MyApp/1.0" \
  -H "Authorization: Bearer token123"

# Response shows the exact request received by server:
# HTTP/1.1 200 OK
# Content-Type: message/http
#
# TRACE /users HTTP/1.1
# Host: api.example.com
# Custom-Header: value
# User-Agent: MyApp/1.0
# Authorization: Bearer token123

# Note: Server may remove sensitive headers like Authorization
# before echoing back the request

// TRACE request for debugging
async function traceRequest(url, customHeaders = {}) {
  try {
    const response = await fetch(url, {
      method: 'TRACE',
      headers: {
        'User-Agent': 'MyApp/1.0',
        ...customHeaders
      }
    });
    
    if (response.ok) {
      const traceData = await response.text();
      return {
        status: response.status,
        contentType: response.headers.get('Content-Type'),
        traceData: traceData
      };
    } else {
      throw new Error(`HTTP ${response.status}: ${response.statusText}`);
    }
  } catch (error) {
    console.error('TRACE request failed:', error);
    throw error;
  }
}

TRACE Security Considerations

Production Security

Disable TRACE in production:

  • Apache: TraceEnable off in httpd.conf
  • Nginx: limit_except GET POST PUT DELETE { deny all; }
  • Node.js: Implement method filtering in your framework
  • Cloudflare: Use WAF rules to block TRACE requests
