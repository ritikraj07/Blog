What are Conditional HTTP Requests?

Conditional HTTP requests are a mechanism that allows clients to make requests that will only be fulfilled if certain conditions are met. This enables efficient caching, reduces bandwidth usage, and improves web performance.

Analogy: Library Book Return

Think of conditional requests like returning a library book:

  • First visit = You borrow a book (full response)
  • Subsequent visits = You check if there's a newer edition (conditional request)
  • No changes = Library tells you your copy is current (304 Not Modified)
  • New edition = Library gives you the updated book (200 OK with new content)

The Problem Conditional Requests Solve

Without conditional requests, browsers would need to download the entire resource every time, even if it hasn't changed. This wastes bandwidth and increases page load times.

Without Conditional
100% Data Transfer
With Conditional
15% Data Transfer (avg)

Key Concepts

Validators

Values that represent the state of a resource, like ETags (entity tags) or Last-Modified dates, used to check if content has changed.

Conditional Headers

Headers like If-None-Match and If-Modified-Since that include validators to conditionally request resources.

304 Not Modified

The HTTP status code returned when a conditional request confirms the cached version is still valid.

How Conditional Requests Work

Conditional requests follow a specific flow between client and server to determine if a cached resource is still valid.

1

Initial Request

Client makes a regular request for a resource. Server responds with the content and validators (ETag, Last-Modified).

GET /styles.css HTTP/1.1
Host: example.com
HTTP/1.1 200 OK
Content-Type: text/css
ETag: "33a64df551425fcc55e4d42a148795d9f25f89d4"
Last-Modified: Wed, 21 Oct 2023 07:28:00 GMT
...
2

Subsequent Request

Client makes a conditional request with the stored validators to check if the resource has changed.

GET /styles.css HTTP/1.1
Host: example.com
If-None-Match: "33a64df551425fcc55e4d42a148795d9f25f89d4"
If-Modified-Since: Wed, 21 Oct 2023 07:28:00 GMT
3

Server Validation

Server compares the provided validators with the current resource state.

  • If validators match: Resource hasn't changed
  • If validators don't match: Resource has changed
4

Server Response

Based on validation, server responds accordingly:

HTTP/1.1 304 Not Modified
ETag: "33a64df551425fcc55e4d42a148795d9f25f89d4"
...

OR

HTTP/1.1 200 OK
Content-Type: text/css
ETag: "a3b72df551425fcc55e4d42a148795d9f25f89e5"
...
/* Updated CSS content */

Performance Impact

Conditional requests dramatically reduce bandwidth usage and improve page load times:

  • Bandwidth savings: Up to 85% for unchanged resources
  • Reduced server load: Fewer full resource deliveries
  • Faster page loads: Less data to download and process
3

Conditional Request Headers

Conditional requests use specific headers to communicate validation information between client and server.

Request Headers (Client → Server)

Header Purpose Validator Used Example
If-None-Match Request content only if ETag doesn't match ETag If-None-Match: "33a64df551425fcc55e4d42a148795d9f25f89d4"
If-Modified-Since Request content only if modified since date Last-Modified If-Modified-Since: Wed, 21 Oct 2023 07:28:00 GMT
If-Match Request only if ETag matches (for safe operations) ETag If-Match: "33a64df551425fcc55e4d42a148795d9f25f89d4"
If-Unmodified-Since Request only if not modified since date Last-Modified If-Unmodified-Since: Wed, 21 Oct 2023 07:28:00 GMT
If-Range Conditional range request ETag or Date If-Range: "33a64df551425fcc55e4d42a148795d9f25f89d4"

Response Headers (Server → Client)

Header Purpose Type Example
ETag Entity tag for resource version Strong/Weak Validator ETag: "33a64df551425fcc55e4d42a148795d9f25f89d4"
Last-Modified Date when resource was last modified Date Validator Last-Modified: Wed, 21 Oct 2023 07:28:00 GMT
Cache-Control Directives for caching mechanisms Cache Directive Cache-Control: max-age=3600, must-revalidate
Vary Indicates which headers affect content Content Negotiation Vary: Accept-Encoding, User-Agent

ETag vs Last-Modified

Feature ETag Last-Modified
Precision High (exact content match) Low (1-second granularity)
Reliability Very reliable Can be inaccurate if clock skew
Generation Server-defined (hash, version, etc.) File system modification time
Storage Overhead Higher (string value) Lower (date value)
Use Cases Dynamic content, frequent updates Static files, infrequent updates

ETag Generation Examples

ETags can be generated in various ways depending on the application:

// Hash-based ETag (common for static files)
ETag = md5(file_content)

// Version-based ETag (for database content)
ETag = "v" + resource_version

// Timestamp-based ETag (simple approach)
ETag = "t" + last_modified_timestamp

// Composite ETag (multiple factors)
ETag = md5(content + metadata)

Advanced Concepts

Strong vs Weak ETags

ETags can be classified as strong or weak validators, indicating how strictly they represent resource changes.

Aspect Strong ETag Weak ETag
Syntax "abc123" W/"abc123"
Precision Byte-for-byte identical Semantically equivalent
Use Cases Static files, exact matches Dynamic content, template changes
Range Requests Fully supported Not supported
Example File content hash Database record version

Cache Validation Strategies

STRONG Strong Validation

Uses both ETag and Last-Modified for maximum accuracy. Most reliable but requires more computation.

ETag: "abc123"
Last-Modified: Wed, 21 Oct 2023 07:28:00 GMT
Cache-Control: max-age=3600, must-revalidate

WEAK Weak Validation

Uses only Last-Modified or weak ETags. Less accurate but simpler to implement.

Last-Modified: Wed, 21 Oct 2023 07:28:00 GMT
Cache-Control: max-age=3600

HEURISTIC Heuristic Expiration

Uses algorithms to estimate freshness when no explicit directives are provided.

# Common heuristic: 10% of age since last modification
 freshness_lifetime = (current_time - last_modified) * 0.1

Implementation Examples

Node.js/Express Implementation

const express = require('express');
const crypto = require('crypto');
const app = express();

// Middleware to generate ETag
app.use((req, res, next) => {
  const originalSend = res.send;
  res.send = function(data) {
    // Generate ETag from response body
    const etag = crypto.createHash('md5').update(data).digest('hex');
    res.set('ETag', etag);
    
    // Check If-None-Match header
    const clientEtag = req.headers['if-none-match'];
    if (clientEtag === etag) {
      return res.status(304).end();
    }
    
    originalSend.call(this, data);
  };
  next();
});

app.get('/api/data', (req, res) => {
  const data = { message: 'Hello World', timestamp: Date.now() };
  res.json(data);
});

Common Pitfalls

  • Weak ETags for critical content: May serve stale content
  • Incorrect date formats: Always use GMT for Last-Modified
  • Over-caching dynamic content: Use appropriate Cache-Control
  • Ignoring Vary header: Can serve wrong content variants
