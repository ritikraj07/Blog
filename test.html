How the Web Works

Client-Server Architecture: The fundamental model of web communication where clients (browsers) request resources from servers (computers storing websites).

Clients

Clients are end-user devices and software that initiate requests:

  • Devices: Computers, smartphones, tablets
  • Software: Web browsers (Chrome, Firefox, Safari)
  • Function: Send HTTP requests and render responses

Servers

Servers are powerful computers that store and serve web content:

  • Function: Receive requests, process them, send responses
  • Content: Store HTML, CSS, JavaScript, images, videos
  • Types: Web servers, application servers, database servers

Complete Request-Response Cycle:

User types URL ‚Üí Browser sends HTTP Request ‚Üí Server processes request ‚Üí Server sends HTTP Response ‚Üí Browser renders HTML/CSS/JS ‚Üí User sees webpage

The Web Toolbox

Internet Connection

The physical/non-physical medium for data transfer between clients and servers.

TCP/IP Protocol Suite

Transmission Control Protocol/Internet Protocol - the foundation of internet communication:

TCP: Reliable, connection-oriented protocol ensuring data arrives completely and in order.

IP: Handles addressing and routing of packets across networks.

Ports: HTTP uses port 80, HTTPS uses port 443.

DNS (Domain Name System)

The internet's phonebook that translates domain names to IP addresses:

DNS Resolution Process:

1. User enters "google.com" 2. Browser checks local DNS cache 3. If not found, queries DNS resolver 4. Resolver queries root DNS servers 5. Root directs to .com TLD servers 6. TLD directs to google.com authoritative server 7. Returns IP address (142.250.185.78) 8. Browser caches IP for future use

HTTP (Hypertext Transfer Protocol)

Application-layer protocol for communication between clients and servers:

HTTP Methods: GET (retrieve), POST (submit), PUT (update), DELETE (remove), HEAD (headers only), OPTIONS (supported methods)

Same Origin Policy (SOP)

Same Origin Policy: A critical security mechanism that restricts how a document or script loaded from one origin can interact with resources from another origin.

Origin Definition

Two URLs have the same origin if they share the same:

  • Protocol (http vs https)
  • Host (domain/subdomain)
  • Port (80 vs 443 vs 8080)

Origin Comparison Examples

Comparison URL Origin Match? Reason
http://store.company.com/dir2/other.html ‚úÖ Same Only path differs
https://store.company.com/page.html ‚ùå Different Different protocol (http vs https)
http://store.company.com:81/dir/page.html ‚ùå Different Different port (80 vs 81)
http://news.company.com/dir/page.html ‚ùå Different Different host (store vs news)

Exam Question Alert: Be prepared to identify whether two URLs have the same origin. Remember: protocol, host, and port must ALL match exactly.

Referrer Policy

Referrer Header: Sent by browsers to tell servers where the request came from (which page linked to the current request).

Referrer Policy: Controls how much referrer information is shared for privacy and security.

Common Referrer Policy Values

Policy Behavior Use Case
no-referrer No Referer header sent at all Maximum privacy
no-referrer-when-downgrade (default) Full URL for HTTPS‚ÜíHTTPS, nothing for HTTPS‚ÜíHTTP Balance security & functionality
origin Only scheme+host (no path) Basic origin information
strict-origin-when-cross-origin (recommended) Full URL for same-origin, origin only for cross-origin, nothing on downgrade Modern default
unsafe-url Always sends full URL (including query parameters) Debugging only (security risk)

Referrer vs Origin Header Difference:

Origin Header: Only contains scheme://host:port

Origin: https://www.youtube.com

Referer Header: Contains full URL including path

Referer: https://www.youtube.com/watch?v=abcd1234

HTTP Headers Detailed Analysis

Request Headers Example

GET /pagead/id HTTP/1.1 Host: googleads.g.doubleclick.net Accept: */* Accept-Encoding: gzip, deflate, br, zstd Accept-Language: en-GB,en;q=0.5 Connection: keep-alive Origin: https://www.youtube.com Referer: https://www.youtube.com/ Sec-Fetch-Dest: empty Sec-Fetch-Mode: cors Sec-Fetch-Site: cross-site User-Agent: Mozilla/5.0 (Macintosh; Intel Mac OS X 10.15; rv:142.0) Gecko/20100101 Firefox/142.0

Header Analysis:

  • Accept: Client accepts any MIME type
  • Accept-Encoding: Supports compressed responses (saves bandwidth)
  • Connection: keep-alive Keep TCP connection open for multiple requests
  • Sec-Fetch-* headers: Modern security headers for fetch requests
  • User-Agent: Identifies browser, OS, and version

How Browsers Load and Render Websites

Step-by-Step Rendering Pipeline

HTML Download ‚Üí HTML Parsing ‚Üí DOM Construction ‚Üí CSS Parsing ‚Üí Render Tree ‚Üí Layout ‚Üí Paint ‚Üí Composite

Detailed Process:

  1. HTML Download & Parsing: Browser downloads HTML and parses it into tokens
  2. DOM Construction: Tokens are converted to nodes, forming DOM tree
  3. CSS Download & Parsing: CSS is parsed into CSSOM (CSS Object Model)
  4. Render Tree: DOM + CSSOM combined into render tree
  5. Layout: Calculate exact position and size of each element
  6. Paint: Fill pixels with colors, images, borders, etc.
  7. Composite: Layers are combined in correct order

HTML DOM (Document Object Model)

DOM: A programming interface for HTML/XML documents that represents the structure as a tree of objects that can be manipulated with JavaScript.

DOM Tree Structure

HTML to DOM Conversion:

<html> <head> <title>My Page</title> </head> <body> <h1>Hello</h1> <p>Welcome!</p> </body> </html>

DOM Tree: document ‚Üí html ‚Üí [head, body] ‚Üí head ‚Üí [title] ‚Üí body ‚Üí [h1, p]

DOM Node Types

  • Document Node: The entire document (root)
  • Element Node: HTML tags (div, p, h1, etc.)
  • Attribute Node: Attributes of elements (class, id, src)
  • Text Node: Text content inside elements
  • Comment Node: HTML comments

DOM Manipulation with JavaScript

Basic DOM Manipulation Methods

1. Selecting Elements:

// By ID document.getElementById('myId'); // By class name (returns HTMLCollection) document.getElementsByClassName('myClass'); // By tag name (returns HTMLCollection) document.getElementsByTagName('div'); // CSS selector (returns first match) document.querySelector('.myClass'); // CSS selector (returns all matches) document.querySelectorAll('p');

2. Modifying Elements:

// Change text content element.innerText = 'New text'; // Change HTML content element.innerHTML = '<strong>Bold text</strong>'; // Change CSS styles element.style.color = 'red'; element.style.backgroundColor = '#000'; // Modify attributes element.setAttribute('class', 'new-class'); element.getAttribute('href'); element.removeAttribute('disabled');

3. Creating/Removing Elements:

// Create new element const newDiv = document.createElement('div'); // Add to DOM parentElement.appendChild(newDiv); // Remove element element.remove(); // Insert before parentElement.insertBefore(newElement, referenceElement);

Virtual DOM

Virtual DOM: A lightweight JavaScript representation of the actual DOM. React creates and maintains this virtual copy to optimize updates to the real DOM.

Why Virtual DOM is Needed

Problem: Direct DOM manipulation is slow because browsers must: 1. Recalculate styles (reflow) 2. Update layout (relayout) 3. Repaint affected elements These operations are expensive for complex applications.

How Virtual DOM Works

State Change ‚Üí New Virtual DOM ‚Üí Diff Algorithm ‚Üí Minimal DOM Updates ‚Üí Real DOM Update

Step-by-Step Process:

  1. Initial Render: Create virtual DOM tree from React components
  2. State Change: User interaction triggers state update
  3. Re-render Virtual DOM: Create new virtual DOM tree
  4. Diffing: Compare new and old virtual DOM trees to find differences
  5. Reconciliation: Calculate minimal set of DOM operations needed
  6. Batch Update: Apply all changes to real DOM in one batch

React Counter Virtual DOM Example:

import React, { useState } from 'react'; function App() { const [count, setCount] = useState(0); return ( <div> <h1>Counter: {count}</h1> <button onClick={() => setCount(count + 1)}> Increment </button> </div> ); }

Virtual DOM Structure (JSON representation):

{ type: "div", props: {}, children: [ { type: "h1", props: {}, children: [ { type: "TEXT_ELEMENT", props: { nodeValue: "Counter: 0" } } ] }, { type: "button", props: { onClick: "setCount(count + 1)" }, children: [ { type: "TEXT_ELEMENT", props: { nodeValue: "Increment" } } ] } ] }

Benefits of Virtual DOM

  • Performance: Minimizes expensive DOM operations
  • Declarative: Describe UI state, not how to update it
  • Cross-platform: Same reconciler for web and native (React Native)
  • Optimized updates: Batches multiple state changes

Common Misconceptions

Misconception 1: Virtual DOM is a browser feature.
Reality: It's a React library implementation.

Misconception 2: Virtual DOM replaces real DOM.
Reality: It's an intermediary optimization layer.

Misconception 3: Only React uses Virtual DOM.
Reality: Vue.js, SolidJS, and others also use similar concepts.

Misconception 4: Virtual DOM solves all performance issues.
Reality: Poor code can still cause performance problems.

Misconception 5: Virtual DOM = Shadow DOM.
Reality: Completely different concepts (Shadow DOM is for encapsulation).

React Fiber Architecture

React Fiber: A complete rewrite of React's core algorithm introduced in React 16. It enables incremental rendering, allowing React to split rendering work into chunks and prioritize updates.

Problems with Stack Reconciler (Pre-Fiber)

  • Synchronous: Once rendering starts, it can't be interrupted
  • Blocking: Large component trees block main thread
  • No prioritization: All updates treated equally
  • Jank: Animations could drop frames

Fiber's Solutions

  • Incremental Rendering: Split work into chunks
  • Priority-based Scheduling: User interactions > background updates
  • Pause/Resume: Can stop work and continue later
  • Error Boundaries: Better error handling

Fiber Architecture Benefits

Key Improvements:

  1. Smoother Animations: High-priority updates don't wait for low-priority work
  2. Better Responsiveness: User interactions are prioritized
  3. Concurrent Features: Suspense, transitions, streaming SSR
  4. Memory Efficiency: Can reuse or abort work

Shadow DOM

Shadow DOM: A browser technology that enables encapsulation of DOM subtrees, allowing components to have private styling and markup that doesn't leak to the main document.

Shadow DOM vs Virtual DOM

Aspect Shadow DOM Virtual DOM
Purpose Encapsulation and scoping Performance optimization
Browser Feature ‚úÖ Native browser API ‚ùå Library implementation
Use Case Web Components, custom elements React, Vue.js, other frameworks
Styling Scoped styles (don't leak out) Global or CSS-in-JS

Creating Shadow DOM:

// Create element with shadow DOM const element = document.createElement('div'); const shadowRoot = element.attachShadow({ mode: 'open' }); // Add encapsulated content shadowRoot.innerHTML = ` <style> p { color: red; } /* Only applies inside shadow DOM */ </style> <p>Hello from Shadow DOM!</p> `; // Add to document document.body.appendChild(element);

Browser controls like <input type="range"> use Shadow DOM internally.

Single Page vs Multi-Page Applications

Definitions

Single Page Application (SPA): A web application that loads a single HTML page and dynamically updates content as users interact. Examples: Gmail, Facebook, Trello.

Multi-Page Application (MPA): A traditional web application where each page reloads completely when navigating. Examples: Wikipedia, Amazon, eBay.

Detailed Comparison

Feature SPA MPA
Page Loading Initial load only, then dynamic updates Full page reload for each navigation
Performance Fast after initial load (like native app) Consistent but potentially slower
SEO Challenging (requires SSR/SSG) Excellent (each page indexable)
Development Complex (routing, state management) Straightforward (traditional web)
Browser History Manual management (History API) Automatic (browser handles)
Offline Support Excellent (Service Workers, PWA) Limited (traditional caching)
Examples React apps, Angular, Vue.js WordPress, Drupal, PHP sites

When to Choose SPA

  • Highly interactive applications (dashboards, social media)
  • Applications requiring offline functionality
  • When you want app-like user experience
  • When you can implement SSR for SEO

When to Choose MPA

  • Content-heavy websites (blogs, news, e-commerce)
  • When SEO is primary concern
  • Applications requiring broad browser compatibility
  • When JavaScript might be disabled

Real-world Scenarios & Practice Questions

Scenario 1: Low-Connectivity News Portal

A startup wants to build a news portal for areas with poor internet. Articles need to load quickly, and users revisit sections frequently.

Question: SPA or MPA? Justify your choice.

Answer: SPA - Can cache articles, implement offline reading, and minimize data transfer after initial load.

Scenario 2: SEO-focused Magazine

An online magazine wants maximum search engine visibility for thousands of articles.

Question: SPA or MPA? Justify your choice.

Answer: MPA - Each article has unique URL, easily indexable by search engines without complex SSR setup.

Scenario 3: Government Portal Accessibility

A government portal must work on older browsers and devices with JavaScript disabled.

Question: SPA or MPA? Justify your choice.

Answer: MPA - Works without JavaScript, compatible with assistive technologies and legacy systems.
